## 一、Hierarchical Agent Architecture 概念整理

### 背景

在 Harness Engineering 的框架下，Agent 層級架構（Hierarchical Agent Architecture）是為了解決單一 AI 模型在處理複雜長任務時容易產生的「上下文迷失」與「邏輯崩潰」問題。

技術主流（參考 2026 年初的相關文獻與實踐）將其拆解為三個核心層級，每一層都有其專屬的 Harness（支架/約束）。

---

### 三層核心架構

#### 1. 策略調度層：總體規劃者（The Planner / Architect）

架構最上層，負責將模糊的人類需求轉化為具體的執行藍圖。

**核心職責**：任務分解（Decomposition）、資源分配、里程碑設定

**Harness 設計**：

| 機制 | 說明 |
|------|------|
| 狀態追蹤器（State Tracker）| 記錄哪些子任務已完成，哪些失敗 |
| 拓撲約束 | 確保任務按邏輯順序（DAG，有向無環圖）執行，防止循環邏輯 |

**輸出**：不直接產出結果，而是產出給下層的指令清單（To-Do List）

---

#### 2. 邏輯管理層：子任務經理（The Manager / Controller）

介於規劃與執行之間，通常按功能模塊劃分（例如：前端經理、資料庫經理）。

**核心職責**：管理一組特定的 Worker、審核 Worker 的產出、決定何時向 Planner 回報

**Harness 設計**：

| 機制 | 說明 |
|------|------|
| 上下文壓縮（Context Compression）| 將 Worker 產生的大量日誌摘要化，只回報關鍵結論給 Planner，防止 Planner 信息過載 |
| 重試邏輯（Retry Policy）| 當 Worker 失敗時，Manager 決定是「換個方法試」還是「上報失敗」 |

**輸出**：對 Worker 的具體動作調用

---

#### 3. 原子執行層：執行員工（The Worker / Executor）

最底層，負責與實際環境（終端機、瀏覽器、編輯器）互動。

**核心職責**：編寫程式碼、運行測試、讀取文件

**Harness 設計（關鍵）**：

| 機制 | 說明 |
|------|------|
| 沙盒化（Sandboxing）| Worker 必須在獨立的、可丟棄的副本環境中操作，防止弄壞主分支 |
| 工具權限控制（RBAC for Tools）| 限制 Worker 只能使用特定的 API 或指令（例如：只能 read 不能 rm -rf）|
| 結構化回報（Structured Handoff）| 強制要求 Worker 以特定格式（如 JSON）回報結果，包含「進度、發現、潛在風險」 |

---

### 層級間的協作機制：對抗與回饋（Feedback Loops）

為了保證質量，現代架構通常會引入「對抗角色」：

**Critic Agent（評論家）**

每一層的產出都會被另一個同級別的 Agent 挑戰。
例如，Worker 寫完代碼後，由 Critic 進行 Code Review，
未通過則直接打回重寫，不驚動上層 Manager。

**Handoff 機制**

當下層任務結束，它必須提供一個「交接包」。
這不是簡單的「做完了」，而是一個包含執行證明（Logs/Tests）的完整報告。

---

### 為什麼這種架構有效？

| 優點 | 說明 |
|------|------|
| 克服「推理疲勞」| AI 模型在單次對話中處理的事情越多，智力下降越快。層級架構讓每個 Agent 只專注於極小範圍 |
| 並行處理 | Planner 可以同時啟動多個 Manager，分別在不同的沙盒環境中並行作業，極大提升效率 |

## 三、Agent 托管在 Microsoft Azure 的話術說明

### 核心論點

> 「個金 Co-Pilot 的三層 Hierarchical Agent 架構，天然對齊 Azure AI Foundry 的托管規格——
> 每一層都有對應的 Azure 服務承接，不是事後硬套，是架構本來就長這樣。」

---

### 三層 × Azure 服務對應

```
┌──────────────────────────────────────────────────────────────────┐
│  Layer 1：策略調度層（Planner）                                   │
│  個金 Co-Pilot 指揮艙                                             │
│  ↕ Azure AI Foundry Agent Service                                │
│  · 統一 orchestration runtime，管理跨 Crew 的 DAG 執行            │
│  · 自動 scaling：案件量暴增時，Foundry 自動擴充 Crew 容量          │
│  · Session 狀態由 Foundry 管理，不需自建 State Tracker DB          │
├──────────────────────────────────────────────────────────────────┤
│  Layer 2：邏輯管理層（Manager）                                   │
│  六大 Pilot Crew 機長                                             │
│  ↕ Azure AI Foundry Hosted Agents（Preview）                     │
│  · 每個 Crew = 一個 Hosted Agent，獨立部署、獨立版本控制           │
│  · 支援 LangGraph / CrewAI 直接上雲，不需 containerization        │
│  · Microsoft Entra ID：每個 Crew 有專屬身份，RBAC 到 Agent 層級   │
├──────────────────────────────────────────────────────────────────┤
│  Layer 3：原子執行層（Worker）                                    │
│  各 Crew 執行單元（XGBoost / Monte Carlo / Claude Vision）        │
│  ↕ Azure 底層服務                                                 │
│  · Azure AI Search：Crew 3 法規 Worker 的向量語義搜尋             │
│  · Azure Cosmos DB：案件資料持久化，Worker 結果寫入               │
│  · Azure Application Insights：每個 Worker 呼叫埋點，全程追蹤     │
└──────────────────────────────────────────────────────────────────┘
```

### 企業安全對應

| 企業需求（銀行合規）| Azure 解法 |
|------------------|-----------|
| 每個 Agent 要有身份，可稽核 | Microsoft Entra ID — 每個 Crew 有專屬 Service Principal |
| 資料不能出台灣 | Private Networking + Azure Taiwan North 區域 |
| 誰呼叫了哪個 Agent，要留紀錄 | Application Insights — 全 Crew 執行軌跡，可查詢 90 天 |
| 不同行員看不同功能 | Azure RBAC — 行員 / 主管 / 維運 三種角色對應不同 API 權限 |

## 四、Azure 托管的好處

### 問題：自己架 vs 托管，差在哪？

把六大 Pilot Crew 放上 Azure AI Foundry Hosted Agents，好處分四個維度：

---

#### 1. 維運成本：從「養機器」變「用服務」

| 自己架（現況）| Azure 托管 |
|------------|-----------|
| 需要一台常駐 server 跑 Node.js | 零台 server，按呼叫次數計費 |
| Agent 掛掉要自己重啟 | Foundry 自動重啟、自動 health check |
| 要自己寫 retry / failover 邏輯 | Foundry 內建 retry policy |
| 黑客松 Demo 當天流量暴增可能掛 | Foundry 自動 scaling，不需預估容量 |

> **白話**：現在 Crew 壞了，你要去 terminal 重啟。上了 Foundry，Foundry 自己顧。

---

#### 2. 安全合規：銀行監理的硬需求

| 監理需求 | 自己架的做法 | Azure Foundry 的做法 |
|---------|-----------|-------------------|
| 每個 Agent 要有身份 | 自己實作 API Key | Microsoft Entra ID，每個 Crew 有 Service Principal |
| 呼叫紀錄要可查 | 自己寫 log 到 DB | Application Insights 自動記錄，可查 90 天 |
| 資料不能外洩 | 自己設防火牆 | Private Networking，流量不走公網 |
| 不同人看不同功能 | 自己實作 RBAC | Azure RBAC 到 Agent 層級，行員/主管/維運各有權限 |

> **白話**：金融監理機關來查，你可以直接開 Azure portal 給他看稽核紀錄，
> 不用臨時撈 log。

---

#### 3. 開發效率：讓工程師專注在業務邏輯

**不上托管的工作量**：
```
寫 Agent 邏輯
+ 寫 orchestration 框架
+ 寫 retry / timeout 處理
+ 寫 health check
+ 寫 logging 串接
+ 寫 scaling 規則
+ 管 Docker / K8s
= 很多非業務邏輯的雜事
```

**上了 Foundry 的工作量**：
```
寫 Agent 邏輯
= 完成
（其他全部 Foundry 處理）
```

> **白話**：一個工程師的工時，原本 30% 在業務邏輯、70% 在維運雜事。
> 上了托管之後，70% 省掉，可以拿去做更多 Crew 的功能。

---

#### 4. 模型管理：混合模型不再是惡夢

個金 Co-Pilot 同時用 Claude Sonnet + Claude Vision，未來可能再加 GPT-4o 備援。

| 問題 | 自己管 | Azure Foundry 管 |
|------|-------|----------------|
| Claude API 金鑰管理 | 分散在各個 .env | 統一在 Foundry Key Vault |
| 模型版本升級 | 要改每個 Crew 的 code | Foundry portal 統一切換 |
| Claude + GPT 混用 | 要管兩家 SDK | 同一個 Foundry SDK |
| 用量監控 | 自己串 dashboard | Foundry 內建 token 用量圖表 |

> **白話**：Azure 是全球唯一同時托管 Anthropic Claude 和 OpenAI GPT 的平台。
> 我們用兩家模型，只需要對一家廠商（Microsoft）。

---

### 一句話總結（給評審用）

> 「托管的好處不是技術炫耀，是讓工程師從維運解放出來專注業務，
> 同時讓銀行監理機關看到可稽核的 AI 治理架構——
> 這才是 AI 在金融業落地的真正門檻。」
