## 一、GitHub Copilot Extensions SDK 對專案幫助評估

### 問題
「用 GitHub Copilot SDK 對專案幫助評估」

### SDK 核心能力

`@copilot-extensions/preview-sdk` — 讓開發者在 GitHub Copilot Chat（VS Code / GitHub.com / GitHub Mobile）建立自訂 Agent，用戶可以用 `@your-agent` 呼叫。

| 功能 | 說明 |
|------|------|
| 請求驗證 | 驗證 GitHub Webhook 簽章 |
| Payload 解析 | TypeScript 型別支援 |
| 串流回應事件 | text / references / confirmation / error |
| `transformPayloadForOpenAICompatibility()` | 可轉成 OpenAI 格式，接任何 LLM |

### 三個應用角度評估

**1. 開發輔助工具（最直接）**
- 建立 `@tcb-copilot` Extension，讓開發團隊在 VS Code 問專案架構問題
- 呼叫現有 `/api/rag-query` 回答「Crew 1 評分邏輯？」「XGBoost 資料流？」
- 實作約 1 天，但只是開發者工具，不是客戶功能

**2. 行員工作輔助**
- 行員在 Copilot Chat 問法規 → Extension 呼叫現有 RAG API → 回答
- 展示微軟生態整合，對 4/22 顧問有加分效果
- 限制：銀行行員實際上不太用 VS Code / GitHub Copilot

**3. 整合進 Microsoft 方案 B**
- Demo 架構圖新增一條線：GitHub Copilot Extension → 現有後端
- 對顧問說：「從 LINE Bot 到 GitHub Copilot，整個流程都在 Microsoft 生態」

### 實作評估

- 磁碟：~50 MB
- 建議：Phase 2（4/22–5/10）中段再做，優先度中等

## 二、Copilot Extension 本質澄清

### 問題
「GitHub Copilot 開發的話是不是有可以用 Claude 或 GPT 模型，跟現在架構好像差異不大？」

### 結論：判斷正確，差異接近零

```
Copilot Extension 架構：

用戶在 VS Code 輸入 @tcb-copilot
  ↓
Extension（薄薄的 webhook 層，SDK 負責驗證+格式）
  ↓ 呼叫任意後端
現有 /api/rag-query → Azure OpenAI GPT-4o
  或
現有 /api/committee-review → Claude Sonnet
  ↓
回傳串流文字給 Copilot Chat
```

Extension 本身不提供任何 AI 能力，只是「入口通道」。

### 與現有架構差異對照

| 項目 | 現有架構 | 加上 Copilot Extension |
|------|---------|----------------------|
| AI 模型 | Claude / GPT-4o | **完全一樣** |
| 業務邏輯 | TypeScript Agent | **完全一樣** |
| RAG 管線 | Azure AI Search | **完全一樣** |
| 不同的地方 | LINE Bot / LIFF 入口 | VS Code Copilot Chat 入口 |
| 新增的東西 | — | GitHub App 驗證 + 串流格式 |

### 最終決策

**直接跳過 Copilot Extension**，把那 1 天時間用在 Phase 2 其他項目。

Gemma 本地化（真正改變資料流向和成本結構）和 Power BI（視覺效果強）的優先度明顯更高。

## 三、實價登錄資料修復與 XGBoost 訓練

### 問題
「fetch_lvpr.py 全部下載失敗：File is not a zip file」

### 根本原因

內政部實價登錄下載 API URL 已更換：

| 舊 URL（失效）| 新 URL |
|-------------|--------|
| `DownloadOpenData?type=zip&fileName=113S1_lvr_land_a.zip` | `DownloadHistory?type=season&fileName=113S1` |

新 URL 返回 `Content-Type: application/octet-stream`，ZIP 內含各縣市個別 CSV（命名規則：`{縣市碼}_lvr_land_a.csv`，台北市=`a`、新北市=`f`）。

### 修正內容（`fetch_lvpr.py`）

1. `BASE_URL` 改為 `https://plvr.land.moi.gov.tw/DownloadHistory`
2. URL 參數改為 `?type=season&fileName={roc_year}S{quarter}`
3. Demo 模式只讀取 `a_lvr_land_a.csv`（台北市）+ `f_lvr_land_a.csv`（新北市）

### XGBoost 訓練結果

```
資料量：147,173 筆（雙北 12 季，112S1～114S4）
行政區：41 個
MAPE：15.24%
命中率（±10%）：43.3%
命中率（±20%）：75.2%
```

特徵重要性 Top 3：
- `district`：49.99%（行政區是最重要特徵）
- `year`：11.47%（時間趨勢）
- `property_age`：9.57%（屋齡）

**額外依賴安裝**：`xgboost`、`joblib`、`scikit-learn`、`brew install libomp`

- 失敗時優雅降級，不影響主鑑估流程

### 技術關鍵：Gemma 4 需使用 Chat API + think:False

| 問題 | 原因 | 解法 |
|------|------|------|
| `/api/generate` 回傳空字串 | Gemma 4 不支援舊式 generate 格式 | 改用 `/api/chat` |
| `/api/chat` content 空白 | 預設開啟 extended thinking 模式，輸出在 `thinking` 欄位 | 加 `"think": false` |
| Python `localhost` 連線失敗 | 系統解析到 IPv6 `::1`，Ollama 只監聽 IPv4 | 改用 `http://127.0.0.1:11434` |
