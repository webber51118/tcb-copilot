## 三、各 CREW 技術定位

### CREW 1：行銷 PILOT（推薦系統）

- **核心**：InstructRec 三層混合架構（arXiv:2305.07001）
- **Layer 1**：Content-Based Filtering — `isEligible()` 資格篩選
- **Layer 2**：Instruction Following — `priorityOrder` 排序（青安 > 國軍 > 一般）
- **Layer 3**：LLM 可解釋性 — Claude 個人化推薦說明
- **優勢**：申請表單天然解決冷啟動問題（比電商隱性行為精確）

### CREW 2：鑑估 PILOT（XGBoost 估價）

- **核心**：XGBoost 點估計 + Monte Carlo GBM 壓力情境
- **流程**：XGBoost → 基準點估（如60萬/坪）→ Monte Carlo 1,000路徑 → P5/P50/P95
- **銀行決策邏輯**：核貸依 P5（最壞情境）而非平均值
- **可解釋性**：SHAP 值輸出各特徵貢獻（對應合庫 AI 治理要求）

### CREW 3：防詐 PILOT（強化規則引擎）

- **現有**：`fraudDetector.ts` 8項規則查核
- **新增**：Python XGBoost / Isolation Forest 異常評分（`fraudScoringService.py`）
- **輸出**：`fraud_score`（0-1）+ `top_risk_factors`（SHAP 前三項）
- **決策**：AWS Fraud Detector 2025/11 已停止接受新客戶 → 改用強化現有規則引擎

## 五、M365 整合策略

### Power BI（行員個案即時輔助）

- **受眾**：行員（非管理層報表）
- **用途**：行員觸發推薦/鑑估/防詐後，Power BI 即時顯示該案件結果
- **三個 Tab**：
  - Tab 1：CREW 1 推薦產品排序 + 月付金試算
  - Tab 2：CREW 2 P5/P50/P95 估價區間 + 核貸上限
  - Tab 3：CREW 3 防詐警示等級 + SHAP 風險因子
- **技術**：Power BI REST API `Datasets PostRows`，Python 推論完成後推送

### Power Automate（兩條自動化流程）

1. **防詐警示通知**：`fraud_score > 0.7` → Power Automate HTTP trigger → **Teams 推播主管**
2. **PDF 批覆書自動化**：推薦完成 → Power Automate 呼叫 `/api/generate-pdf` → 存至 **SharePoint**

### 9.2 問題分析

#### 問題 1：資料收集順序違反客戶心理

目前流程：貸款類型 → 年齡/職業/收入 → 房屋資訊 → 推薦

客戶第一個問題通常是「我能借多少？」，但系統要他先說職業、年齡，才到金額。客戶會感覺「在被審查」，中途放棄率高。

**建議順序**：
```
想借多少（金額）→ 貸款類型（金額決定）
→ 用途（首購/換屋/週轉）→ 職業+收入（資格確認）
→ 房屋資訊（若需要）→ 推薦
```

#### 問題 2：推薦卡缺少法遵必要聲明

金管會規定：金融商品推薦必須揭露以下：
- 利率非最終核定利率，依實際核貸為準
- 貸款前請詳閱相關契約
- 「本試算結果不構成貸款承諾」

目前卡片底部僅有「本結果由 AI 模擬，實際核貸依行員審查為準」——**不符合金管會規範，上線前必須修正**。

#### 問題 3：APPLY_DONE 之後客戶不知道下一步

客戶送出申請後，收到 PILOT CREW 結果卡，但接下來呢？流程斷掉了。

客戶實際問題：
- 「我需要帶什麼文件去分行？」
- 「幾天會有消息？」
- 「行員會聯絡我嗎？」

#### 問題 4：語音結果確認按鈕邏輯有問題

```
✅ 確認，顯示推薦方案 → 發送文字「語音確認」
```

`conversationStateMachine` 若無「語音確認」指令處理，客戶按下去會沒有反應。需確認狀態機有對應路由。

#### 問題 5：Session 為 in-memory，中途跳出後無法繼續

客戶填到一半離開（常見），隔天回來會從頭開始。伺服器重啟後全部歸零。對銀行業務來說這是**重大問題**——客戶要重填容易流失。

### 9.3 優化建議（優先順序排列）

#### P0：必須修正（合規 + 體驗斷點）

1. **加入法遵聲明到所有推薦卡**
   - 「試算結果不構成貸款承諾，實際核貸條件依行員審查為準」
   - 「年利率計算基準：指標利率＋加碼，依核貸當日公告為準」

2. **修正「語音確認」指令到狀態機**
   - `conversationStateMachine` 新增：`'語音確認'` → 觸發 RECOMMEND 推薦流程

3. **APPLY_DONE 後補送後續指引訊息**
   - 審核時程說明（3 個工作天）
   - 對保所需文件清單
   - 「查詢申請進度」入口指引

#### P1：高業務價值（轉換率 + 客戶體驗）

4. **申請進度查詢功能**
   - 客戶輸入「查詢進度」，根據 session `applicationId` 回查 `/api/applications`
   - 推送簡易狀態卡（案件編號 / 狀態 / 預計完成日）

5. **預約分行服務按鈕**
   - 推薦卡新增「📍 預約最近分行」按鈕，連到合庫官網預約頁
   - 這是最直接的線下轉換入口

6. **行員主動查詢模式**
   - 行員輸入「行員查詢 APP-XXXX」可拉取案件摘要
   - 減少行員切換後台系統次數

#### P2：差異化功能（競爭優勢）

7. **月還款試算互動卡**
   - 推薦後立即推「試算調整卡」，Quick Reply 讓客戶直接調整年限/金額
   - 即時更新月付金，不需要重來
   - 在 LINE 裡做轉換率高

8. **不動產市場資訊主動推播**
   - 每週主動推給詢問過房貸的用戶：台北市近期成交均價、利率走勢
   - 保持黏著度，競爭對手（玉山、台北富邦）LINE Bot 尚未做好

9. **以房養老獨立流程**
   - 現在混在一般房貸流程，目標客群（60歲以上）使用習慣完全不同
   - 需更大字體、更少選項、更多說明
   - 建議獨立一條更慢、更詳細的對話流程
