### Crew 1 徵審模型（信用評分）

**資料來源**：MyData 介接 + 申請表單

| 特徵維度 | 具體變數 |
|---------|---------|
| People（品格）| 信用分數、逾期次數、帳戶開立年數 |

### 為什麼 XGBoost 適合金融場景

XGBoost 內建 Feature Importance（搭配 SHAP 值），每筆預測可輸出：
> 「屋齡貢獻 -12%、行政區貢獻 +23%、坪數貢獻 +8%⋯」

對應合庫 AI 治理**透明性/可解釋性**要求——決策邏輯可書面化，批覆書可直接印出各特徵影響分數。

## 三、Monte Carlo 用意白話解釋

### 問題場景

```
XGBoost 預測：每坪 60 萬（點估計）
銀行問題：最壞情況能跌到多少？我能放幾成？
```

XGBoost 本身答不了這個問題——它只給一個數字，不給風險區間。

### Monte Carlo GBM 怎麼解決

模型對房價走勢模擬 1,000 條路徑：

```
每條路徑 = 房價 × exp( (μ - σ²/2)×Δt + σ×√Δt×Z )
Z ~ N(0,1) 隨機抽樣
```

跑完 1,000 次後取三個數字：

| 輸出 | 意義 | 銀行用途 |
|------|------|---------|
| **P95（樂觀）** | 95% 情境下房價上限 | 估算最大增值空間 |
| **P50（中位）** | 最可能成交區間 | 主要鑑估參考值 |
| **P5（悲觀）** | 5% 情境下房價下限 | **放貸依據** |

### 銀行實際決策邏輯

```
XGBoost → 基準點估 60萬/坪
Monte Carlo → P5 = 52萬/坪（壓力情境）

核貸金額 ≤ P5估值 × 核貸成數（例如7成）
→ 52萬 × 0.7 = 36.4萬/坪 × 坪數 = 核貸上限
```

壓力情境保護銀行：即使房市下行到最壞 5% 情境，擔保品仍能覆蓋貸款。

### Demo 說法

> 「XGBoost 告訴你現在值多少，Monte Carlo 告訴你最壞值多少——銀行放貸看的是最壞情境，不是平均值。」

## 四、監督式學習 + 梯度提升白話解釋

### 監督式學習

**白話**：給答案讓機器自己找規律。

```
你給機器看 14 萬筆資料：
  這間房：60坪、大安區、屋齡5年  → 每坪 80 萬
  那間房：20坪、文山區、屋齡30年 → 每坪 35 萬
  ...
機器學會：大安區+新屋→高價、文山區+老屋→低價
之後給新房子 → 它就能預測價格
```

**關鍵**：一定要有「正確答案」（label）才能訓練。

### 梯度提升

**白話**：一群弱學生輪流訂正前一個人的錯誤。

```
第 1 棵樹：預測 60 萬（實際 80 萬，差了 20 萬）
第 2 棵樹：專門學哪些情況差 20 萬 → 修正 +12 萬 → 72 萬
第 3 棵樹：專門學哪些情況還差 8 萬 → 修正 +6 萬 → 78 萬
... 繼續疊加直到誤差收斂
最終預測 = 所有樹的結果加總
```

「梯度」意思：每棵新樹都往「最能減少誤差的方向」學，不是隨機猜。

### 層級關係

```
監督式學習（大框架）
  └─ 決策樹（單棵）
       └─ 隨機森林（很多樹，各自獨立）
            └─ 梯度提升（很多樹，後者修正前者）
                 └─ XGBoost（梯度提升的最佳化實作）
```

### Demo 說法

> 「XGBoost 屬於監督式學習——用 14 萬筆有成交紀錄的實價登錄訓練，機器學到台灣房市的定價規律。內部用梯度提升：100 棵樹輪流修正誤差，最終預測精度達到 RMSE < 8%。」

## 五、InstructRec 論文解析（arXiv:2305.07001）

**論文標題**：Recommendation as Instruction Following: A Large Language Model Empowered Recommendation Approach
**作者**：張俊傑等，中國人民大學 + 騰訊 WeChat
**發表**：2023年5月

### 核心問題

傳統推薦系統靠**猜**：
```
觀察你買了什麼 → 推測你要什麼
問題：你想要的跟買過的不一定一樣（冷啟動問題）
```

InstructRec 的想法：**讓用戶直接說出來，LLM 執行指令。**

### 三維指令格式

每條推薦指令由三個維度組成：

| 維度 | 類型 | 說明 |
|------|------|------|
| **Preference（偏好）** | 隱性 P1 / 顯性 P2 | 用戶長期喜好（購買紀錄 / 直接陳述） |
| **Intention（意圖）** | 模糊 I1 / 明確 I2 | 這次想要什麼（「給兒子禮物」/「iPhone 13」） |
| **Task Form（任務型式）** | T0/T2/T3 | 篩選 / 候選生成 / 重新排序 |

### 論文方法

1. 設計 39 個粗粒度指令模板
2. 用 GPT-3.5 自動生成 252,000 筆細粒度個人化指令
3. Fine-tune 3B Flan-T5-XL → InstructRec
4. 結果：在序列推薦、商品搜索、個人化搜索全面超越 GPT-3.5 零樣本

### Co-Pilot 的天然對應

**銀行場景的優勢：申請表單 = 用戶主動給出完整指令（比電商隱性行為精確 100 倍）**

| InstructRec 概念 | Co-Pilot 對應 |
|-----------------|--------------|
| Preference 隱性（P1） | 年齡、職業、收入 → `isEligible()` 資格篩選 |
| Preference 顯性（P2） | 貸款用途（首購自住/投資/周轉） |
| Intention 明確（I2） | 「我要借500萬、還30年」— 申請表單直接填 |
| Task Form 排序（T3） | `priorityOrder`：青安 > 國軍 > 一般 |
| LLM Rationale | Qwen2.5 動態生成推薦說明（Layer 3） |

### 混合架構三層說明

```
Layer 1 — Content-Based（Netflix 冷啟動解法）
  用戶屬性（年齡/職業/收入）→ isEligible() 資格篩選 → 候選產品池
         ↓
Layer 2 — Instruction Following（InstructRec 核心）
  用戶意圖（金額/用途/年限）→ priorityOrder 排序 → 主推產品
         ↓
Layer 3 — LLM Explanation（InstructRec 最大優勢）
  Qwen2.5 → 根據用戶條件 → 個人化推薦說明
```

### Demo 說法（完整版）

> 「我們的推薦引擎實作了 InstructRec 論文的核心概念。
>
> 傳統推薦系統靠猜——看你過去買什麼。
> InstructRec 讓用戶直接下指令：申請貸款時填的表單，
> 就是在告訴系統「我要什麼、我有什麼條件、我的目標是什麼」——
> 比 Netflix 的隱性行為推薦精確得多。
>
> 第一層是 Content-Based Filtering——
> 跟 Netflix 冷啟動解法一樣，用用戶的靜態屬性篩選候選產品。
> 銀行的優勢是申貸必填欄位天然解決了冷啟動問題。
>
> 第二層是 Instruction Following——
> 用戶填表的動作本身就是給系統下指令：
> 「我要借 500 萬、首購自住、還 30 年」——比 Netflix 的隱性行為推薦更精確。
>
> 第三層是 LLM 可解釋性——
> InstructRec 最大的優勢。Qwen2.5 根據每個客戶的具體條件即時生成推薦說明，
> 同一個產品對不同客戶說不同的理由，這是傳統規則引擎做不到的。
> 也正好對應合庫 AI 治理的透明性與可解釋性要求。」

### M365 + Copilot 整合架構（解法）

```
客戶端 (LINE Bot)
    ↓
【Copilot Studio】← 這就是 Agent Harness（直接對應評分「使用Copilot」）
    ↓ 觸發
【Power Automate】← 工作流程自動化
    ├─→ 觸發「台語語音辨識 Agent」(Azure Speech)
    ├─→ 觸發「演算法鑑估 Agent」(XGBoost + Monte Carlo ML 模型)
    └─→ 觸發「InstructRec 推薦 Agent」
    ↓ 輸出結果
【Power BI Embedded】← 行員審核儀表板
    + Teams Adaptive Card 通知行員
    ↓
行員在 Teams 內直接審核/核准（不需另開系統）
```

### 各 M365 工具對應用途

| M365 工具 | 專案中具體用途 | 量化效益 |
|----------|--------------|---------|
| **Copilot Studio** | LINE Bot Agent 框架，取代純 Claude API | 直接對應評分「使用Copilot」|
| **Power Automate** | 申請觸發 → 自動建案 → 鑑估 → 通知行員 | 自動化，節省 40% 手動流程 |
| **Power BI** | 行員儀表板：鑑估視覺化、推薦命中率、核准率趨勢 | 審核從 45 分鐘 → 15 分鐘 |
| **Microsoft Teams** | 鑑估結果以 Adaptive Card 推送，一鍵核准/退件 | 行動審核，不需另開系統 |

### 三個功能對應的 M365 路徑

**1. LINE 推薦系統**
- 前端：LINE Bot（客戶面）
- 後端：Copilot Studio Agent（Microsoft 生態）
- 輸出：Power BI 推薦命中率報表給主管看

**2. 台語模型**
- 輸入：Azure Speech Service（M365 AI 服務之一）
- 理解：自訂台語 LLM
- 亮點：台語 → 普通話 → Copilot Studio 理解，pipeline 本身即技術差異化

**3. 演算法鑑估**
- 觸發：Power Automate 自動呼叫 ML API
- 展示：Power BI Embedded 視覺化（房價預測區間、Monte Carlo 路徑圖）
- 行員端：Teams Adaptive Card 一鍵確認鑑估值
