```
final_score = (金額風險 × 0.3) + (行為模式 × 0.5) + (0.2 × (1 - 調查因子))
```

## 三、Gemini 補充意見 — Human-in-the-Loop 三機制

### 機制一：`input_mode` 強制人工確認節點

AI 在執行關鍵 Task（生成批覆書）前強制跳出確認介面，人類可**即時修正** AI 草稿參數。

**個金 Co-Pilot 落地設計**：
```
AI 產出批覆書草稿
    ↓
[強制介入點] 行員調整核准金額 / 利率 / 附加條件
    ↓
確認後發送 LINE 通知客戶
```
對應位置：`AdminCaseDetailPage` 的「行員最終決策」區塊，現僅有核准/婉拒二元選擇，需新增參數調整表單。

### 機制二：異常路由（Escalation Routing）三級分派

```
防詐分數 ≤ 40   → Level 1：AI 建議核准，行員一鍵確認
防詐分數 41–70  → Level 2：標示「需人工審查」，指派資深行員
防詐分數 > 70   → Level 3：凍結案件，強制主管介入 + 紅色警告
```

對應位置：`AdminCasesPage` 需新增分級視圖，`AdminDashboardPage` 新增「待人工審核」獨立區塊。

### 機制三：Reviewer Portal（獨立人工審核入口）

AWS 有獨立 `/reviewer` Portal 接收 AI 標記案件。個金 Co-Pilot 最簡落地：在案件列表新增 `ai_flagged` 狀態，不需新增頁面。

### 三機制實作優先序

| 機制 | 實作難度 | 黑客松衝擊力 | 建議時程 |
|------|---------|------------|---------|
| 異常路由三級分層 | 低 | ⭐⭐⭐⭐⭐ | 立即實作 |
| `input_mode` 批覆書參數修正 | 中 | ⭐⭐⭐⭐⭐ | 黑客松前 |
| 獨立 Reviewer Portal | 高 | ⭐⭐⭐ | 黑客松後 |

### 三痛點 × 三演算法層（Gemini + Claude 聯合設計）

#### 痛點一：信用小白冷啟動 → Semantic Search + Knowledge-Based

**商業痛點**：年輕新客戶無歷史數據，傳統系統群發罐頭廣告，導致高封鎖率。

**演算法**：LLM 將自然語言轉為 User Vector，與 Product Vector 做 Cosine Similarity。

```
用戶說：「剛出社會想買房，薪水不多」
    ↓
LLM 解析 → User Vector { life_stage: "新鮮人", goal: "購屋", income_level: "低" }
    ↓
Cosine Similarity vs Product Vectors
    ↓
青安貸款 0.92 ✓ > 國軍方案 0.41 > 一般房貸 0.38
```

**黑客松亮點**：不填任何表單，一句話精準推薦。

---

#### 痛點二：隱性資金需求 → Sequential Recommendation + Collaborative Filtering

**商業痛點**：客戶問的是表面問題，背後藏著買房/成家/留學等大額需求，銀行錯失商機。

**演算法**：Attention Mechanism 加權對話序列，協同過濾比對同溫層行為模式。

```
t=1 「想了解房貸利率」    → 意圖向量 A
t=2 「婚後想買三房」      → 意圖向量 B（近期，權重高）
t=3 「現在租房兩萬」      → 意圖向量 C（最新，最高權重）
    ↓ Attention 加權
    ↓ Collaborative Filtering 模式比對
→ 歷史相似序列客群 → 最終申請青安貸款
    ↓
主動推播：「您似乎正在規劃成家，青安貸款適合您」
```

**黑客松話術**：「比客戶更懂客戶——系統能看見客戶的金融人生旅程（Financial Life Journey）中的隱性信號。」

---

#### 痛點三：法規複雜選擇障礙 → Two-Tower Model + Rule Engine

**商業痛點**：金融產品條款繁雜，推薦了客戶無法申辦的產品會引發客訴。

**演算法**：Two-Tower Model（軟預測）疊加 Rule Engine（硬合規過濾）。

```
Tower 1（User Encoder）    Tower 2（Product Encoder）
年齡、職業、收入 ──→ 點積 ←── 利率、成數、資格條件
負債比、貸款用途              合規標籤、適用族群
    ↓
相關性分數（0.0–1.0）
    ↓
Rule Engine 硬過濾
  IF 用途=商辦 → 排除 72-2 管制方案
  IF 職業≠軍公教 → 排除國軍方案
    ↓
最終推薦（軟分數 AND 硬規則）
```

**黑客松話術**：「我們的推薦系統有深度學習的『軟預測』，還疊加了金融合規的『硬規則』——推薦的產品一定能申辦。」

## 六、本地 LLM 模型替換：Gemma 4 → Qwen2.5:14b

### 評估背景

比照 AWS 保險理賠 EKS 專案的本地推論方案（Ollama + Qwen2.5），評估是否將現有 Gemma 4 替換為 Qwen2.5。

### 替換原因

| 維度 | Gemma 4 | Qwen2.5:14b |
|------|---------|-------------|
| 繁體中文能力 | 普通 | 強（阿里訓練） |
| 金融推理 | 普通 | 有金融微調版 |
| think flag 問題 | 需要 `think:false` | 無此問題 |
| 本地推論速度 | 較慢 | 較快（同硬體）|

### 修改範圍

| 檔案 | 修改內容 |
|------|---------|
| `xgboostValuationService.py` | `model` 預設值 `"gemma4"` → `"qwen2.5:14b"`，docstring 更新 |
| `core/app.py` | explain endpoint docstring 更新 |
| `ValuationPage.tsx` | 前端 badge 顯示 `Gemma 4` → `Qwen2.5`，註解更新 |
| `valuationClient.ts` | JSDoc 註解更新 |
| `agentMonitorStore.ts` | AgentName type + 初始清單 `Gemma4本地AI` → `Qwen2.5本地AI` |
| `valuateXgboost.ts` | `recordAgentCall` 名稱同步更新 |

### 切換指令

```bash
ollama pull qwen2.5:14b
```

其餘程式碼無需修改，API 格式與 Gemma 4 完全相同（均為 Ollama `/api/chat`）。
