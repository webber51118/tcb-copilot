## 二、推薦系統現況分析

### 現有系統：規則引擎，不是推薦系統

`recommendationEngine.ts` 的核心是固定優先權陣列：

```typescript
const priorityOrder = [
  REVERSE_MORTGAGE,    // 以房養老
  MILITARY_HOUSING,    // 國軍輔導
  YOUNG_SAFE_HOME,     // 青安
  NEXT_LOAN,           // Next貸
  MILITARY_CIVIL,      // 軍公教
  ELITE_LOAN,          // 優職優利
];
```

**問題**：30 歲公務員首購 和 62 歲退休族，通過 `isEligible()` 後拿到相同排序邏輯。
**根因**：`priorityOrder` 是全局靜態排序，不反映個別用戶的最適配度。

### 與 Netflix 等主流系統比較

| 維度 | 現有系統 | Netflix/電商 |
|------|----------|-------------|
| 演算法類型 | 規則引擎 | CF + 內容過濾 + 深度學習混合 |
| 資料來源 | 當下填寫表單 | 歷史行為（點擊、停留、評分） |
| 個人化程度 | 條件相同 → 推薦相同 | 同條件不同人 → 推薦不同 |
| 可解釋性 | 完全透明 | 黑盒（難以解釋） |

## 三、推薦系統合理升級設計

### 單品類升級：Weighted Scoring 取代 priorityOrder

```
Score = 利率競爭力(30%) + 職業專屬度(25%) + 用途契合度(20%) + 額度適配(15%) + 年齡適配(10%)
```

動態評分讓「30 歲公務員首購」自然得到青安高分，不需寫死順序。

### 跨品類升級：生命週期套餐引擎

**五大人生階段**：
| Life Stage | 核心信號 | 主推套餐 |
|------------|---------|---------|
| 軍公教特殊群體 | 職業=軍/公/教 | 專屬優惠產品 |
| 首購成家族 | 年齡28-40 + 首購 | 青安 + 壽險 + 御璽卡 + CoBaby |
| 資產累積期 | 年齡35-50 + 已有房 | 信貸 + 基金 + 回饋卡 |
| 子女教育期 | 40-55歲 + 有小孩 | 教育基金 + 信託 + 保險 |
| 退休規劃期 | 55歲+ | 以房養老 + 長照保險 + 安養信託 |

**套餐設計原則**：
- 每套有一個「故事」（首購族完整方案 / 退休安養方案 / 軍人全方位方案）
- Budget Constraint：月總承諾 ≤ 月收入 × 50~65%
- `productCatalog.ts` 的 `crossSell` 欄位可作為起點升級

## 四、論文研究：推薦系統學術依據整理

### 論文一：Implicit Feedback CF（隱性反饋協同過濾）
**Collaborative Filtering for Implicit Feedback Datasets**
Hu, Koren & Volinsky — IEEE ICDM 2008

- 核心概念：點擊/瀏覽視為正向偏好指示，但加入「信心加權」處理不確定性
- 場景對應：LINE Bot 點擊行為 → 信心加權矩陣分解
- **適用性評估：最適合現階段**，概念直接可實作，不需大量訓練資料

**信心加權在本系統的應用**：
```
點擊「青安介紹頁」1 次  → 信心=低，興趣信號弱
點擊「青安月付試算」     → 信心=中，意圖更明確
詢問「青安寬限期怎麼算」 → 信心=高，幾乎確定有意願
```

---

### 論文二：SASRec（序列型自注意力推薦）
**Self-Attentive Sequential Recommendation**
Kang & McAuley — ICDM 2018

- 核心概念：用 Self-Attention 捕捉行為序列的動態變化，識別哪些歷史行為對預測下一行為最相關
- 在稀疏和密集資料集上優於 MC 和 RNN，效率比 CNN/RNN 快一個數量級

**⚠️ 現階段適用性分析（結論：不適合）**：
| 前提條件 | 理想場景 | 本系統實際 |
|---------|---------|-----------|
| 物品數量 | 百萬級 | **~30 個** |
| 序列長度 | 數百個行為 | **3-5 個點擊** |
| 訓練資料量 | 億級 | **目前為零** |
| 互動頻率 | 每天數十次 | **每月 1-3 次** |

**結論**：Attention mechanism 在 30 個 item 上學不到有意義的 pattern，結果會退化成接近隨機。應列為**6個月後升級路徑**，不是現階段實作目標。

---

### 論文三：BERT4Rec（雙向 Transformer 序列推薦）
**Sequential Recommendation with Bidirectional Encoder Representations from Transformer**
Sun et al. — CIKM 2019

- 核心概念：用雙向 Transformer 的 Self-Attention 建模行為序列，透過 Cloze 任務預測遮罩項目
- 在多資料集上持續優於各種最先進的序列推薦模型
- **適用性**：與 SASRec 相同問題，現階段 item 數量和資料量不足

---

### 論文四：InstructRec（指令驅動推薦）
**Recommendation as Instruction Following: A Large Language Model Empowered Recommendation Approach**
ArXiv 2023

- 核心概念：用戶以自然語言表達偏好（填表 = 下指令），LLM 映射到推薦結果，完美解決冷啟動
- **適用性：最適合新客戶場景**，不需歷史資料，填表即啟動

---

### 論文適用場景對照

| 客戶類型 | 演算法 | 論文依據 | 現階段可實作 |
|---------|--------|---------|------------|
| 新客戶（無任何紀錄）| InstructRec Zero-shot | ArXiv 2023 | ✅ 可實作 |

| 未來資料足夠後 | SASRec 序列推薦 | ICDM 2018 | 🔮 Phase 3 升級 |

## 七、InstructRec 演算法深度解析

### 核心思想：推薦 = 執行指令

傳統推薦系統問的是：「這個用戶過去喜歡什麼？」
InstructRec 問的是：「這個用戶現在告訴我他要什麼？」

### 運作流程

```
用戶輸入（自然語言）
    ↓
指令格式化（Instruction Template）
    ↓
LLM 理解意圖 + 映射產品
    ↓
排序輸出 + 理由說明
```

### 三種指令類型

論文定義用戶偏好有三個維度：

| 維度 | 說明 | 本系統對應 |
|------|------|-----------|
| **偏好（Preference）** | 我喜歡/需要什麼特性 | 「我要利率低的」「我要寬限期」 |
| **意圖（Intention）** | 我這次的目的 | 「我要買人生第一間房」 |
| **情境（Context）** | 當下的背景條件 | 「30歲、公務員、月薪5萬」 |

三個維度合在一起就是一條「指令」，LLM 根據指令選出最符合的產品。

### 與傳統 CF 的根本差異

```
傳統協同過濾（CF）：
  用戶A 喜歡X → 找到也喜歡X的用戶B → 推薦B喜歡的Y給A
  前提：需要大量歷史行為資料

InstructRec：
  用戶A 說「我需要X條件的產品」→ LLM 直接推理 → 輸出Y
  前提：只需要現在這句話
```

### 論文的技術實作

原始論文的做法：

1. **建立指令模板**：把用戶資訊結構化成標準 prompt 格式
2. **Fine-tune LLM**：在大量「指令→推薦結果」的訓練對上微調模型
3. **產品索引**：每個產品有詳細的自然語言描述（不是 ID）
4. **生成式推薦**：LLM 直接生成推薦結果，不是從清單挑選

### 本系統的實用版本（不需 fine-tune）

原論文需要 fine-tune，但本系統有 Claude API，可以用 **zero-shot prompt engineering** 達到同等效果：

```
System Prompt：
你是合庫個金推薦引擎。以下是所有產品說明：
- 青安：首購、30歲以下加碼、利率1.775%起...
- 國軍輔導：現役/退役軍人、額度最高600萬...
- 以房養老：60歲以上、將房產換現金流...
[其他產品]

User Prompt：
客戶資料：
- 年齡：32歲
- 職業：公務員
- 月收入：52,000
- 用途：首購自住
- 期望額度：800萬

請推薦最適合的前3個產品，並說明原因。
```

這就是 InstructRec 的精神——**填表資料直接成為指令，LLM 做推理，不需要任何歷史資料**。

### 為什麼適合銀行場景

| 優點 | 說明 |
|------|------|
| 冷啟動解決 | 新客戶沒有任何紀錄也能推薦 |
| 可解釋性 | LLM 可以說明「因為你是公務員首購，所以推青安」 |
| 合規友善 | 推薦邏輯透明，可供審計 |
| 彈性高 | 新產品只需更新 prompt，不需重新訓練 |
| 語意理解 | 能處理「我想買房但預算有限」這種模糊輸入 |

**最大弱點**：推薦品質上限取決於 prompt 設計和產品說明文件的品質，而不是資料量。

### 演算法選用策略

| 現在 | 6 個月後 | Phase 3 |
|------|---------|---------|
| InstructRec（zero-shot prompt） | + Implicit Feedback 信心加權 | + SASRec 序列推薦 |
